# Writeup

## What I built

HospitalMatch ranks hospitals by real CMS quality outcomes for a patient chosen condition. A patient describes symptoms, gives insurance and location, and gets nearby hospitals ranked on actual Medicare outcome measures, each with a plain explanation grounded in the real numbers.

## The user and the problem

User: a patient or care navigator choosing among hospitals. The raw CMS data is opaque. The headline star rating is missing for 41 percent of hospitals, and the underlying measures live across separate files keyed only by Facility ID. A normal person cannot turn that into a decision.

## How the dataset is used

The core is a real join, not a wrapper. I pull six CMS files (general information plus complications and deaths, unplanned visits, timely and effective care, infections, HCAHPS) and join them on Facility ID into one store of about 800k measure rows. Each condition maps to the specific measures that matter for it, for example knee surgery maps to the hip and knee complication rate, the 30 day readmission rate, and patient experience.

## Scoring and the knowledge graph

Scoring is percentile based against the full national pool of every hospital that reports each measure, which is robust to outliers. Direction is handled per measure, so for death and readmission lower is better and the percentile flips. Weights are renormalized when a measure is missing, and a hospital is never imputed. Missing measures are excluded and the reason is decoded from the CMS footnote, so the tool says why data is absent instead of hiding it. The ranking runs over a NetworkX knowledge graph of Condition to Measure to Hospital, so a rank is a traversal with provenance rather than a single opaque number.

## Explanations

Each ranked hospital gets a two sentence explanation. With an API key it is generated by Claude grounded only in that hospital real numbers. Without a key it falls back to a deterministic template built from the same numbers, so the app runs with zero setup. A grounding check confirms explanations contain no number that is not in the data.

## Tradeoffs

Insurance is collected and shown but not network verified, because the CMS data has no plan network field. The UI states this and tells the user to confirm with their insurer rather than fake it. Symptoms are mapped to four conditions by keyword, since those are the conditions with strong real measures. Location filters by state. The store is cached in process and warmed on startup, because a per request pull of 800k rows was slow.

## What I would improve with more time

Real insurance network data, a geocoded distance filter instead of state, more conditions, and a verifier model in a different family to adversarially check explanations before display.

## LLM assistance

AI assistance shaped this build. I used a multi model orchestration plan to route work by task, used Claude to write most implementation and tests, and verified every phase against live CMS data myself, including independent join spot checks and adversarial grounding probes. The runtime explanation feature calls Claude Sonnet with a template fallback.
